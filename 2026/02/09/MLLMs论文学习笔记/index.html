MLLMs论文学习笔记

Crab: A Unified Audio-Visual Scene Understanding Model with Explicit Cooperation

问题背景

现在音频-视觉的多模态大模型能完成很多任务，比如：

  • 时间定位： 在视频中找出某事件发生的时间段;
  • 空间定位： 找出声音来源、动作主体的位置；
  • 时空推理： 推理事件的因果关系、动作意图；
  • 像素级理解： 像素级的图像分割、目标检测。

显然这些任务之间是有 内在联系 的，比如时间定位、空间定位可能需要时空推理提供理解上的帮助，像素级分割需要先大致进行时间空间定位等。而现在简单的多任务联合训练不能很好地利用这种内在联系，有待改进。

让我们首先来看看什么是“简单的多任务联合训练”。最原始的阶段，多任务是通过使用不同的输出头来实现的，同一个 backbone，给每个任务一个输出头，如下图所示：

这么做的问题非常明显，如果不冻结 backbone，那么 loss 会直接打到 backbone 上去，随着 backbone 的增大（比如 LLM）慢慢调不动了；如果冻结 backbone，那实际上只在训练一个几层的小输出头，效果很差。

这部分基本上是前 LoRA 时代的历史，在这个时代多任务联合训练是怎么进行的呢？如果不冻结 backbone，那可能还有点意义，至少不同任务都能反馈到 backbone 模型上（只要能训得动就行）；如果冻结了 backbone，共享表示就全部被冻结了，数据之间没有交互，任务没法协同。

此后，LoRA 被发明出来，通过在 LLM 内部开辟旁路的方式，用低秩矩阵来“改一点点大模型”，这让大模型微调成为可能，但是还是没有很好地解决多任务联合训练的问题。有的方法只用一组 LoRA，所有任务都是调这同一组 LoRA，不同的任务会将模型扯向不同的方向，loss scale 等问题也很成问题，最终导致训练甲任务后乙任务表现下降——这就像是一个脑力有限的人同时学八门课，把自己弄糊涂了。

而另一个方案是给每个任务一组单独的 LoRA，互不通用，这当然解决了不同任务对齐的问题，但是各个任务之间也无法互相借鉴，很大程度上丧失了联合训练的意义——八个人每人学一门课，互不交流，“数学”就没法为“物理”提供帮助了。

创新点

本文有两个主要贡献：设计 AV-UIE 数据集，以及新型的多头 LoRA 机制。首先看 AV-UIE 数据集：

之前的数据集，不管是时间定位、空间定位还是时空推理等，其回答都是简单的。如询问“第一个响起的乐器的左侧的乐器是什么？”，其回答可能只有“大提琴”三个字；如果是时间定位，可能只有简单的“3-6 秒”几个字。作者基于已有的此类数据集进行扩展，利用 LLM 进行扩写和人工修改，在保留原来回答的基础上增加了显式的其他任务的信息，比如：

通过改造数据集，在输入阶段就提供了显式的多任务联合的监督信号。

接下来看 LoRA 的设计。和一般的多模态大模型一样，Crab 模型分为前端和大模型两部分，前端采用了 Clip 作为视觉编码器，采用 BEATs 作为音频编码器，用 Q-Former 查询视觉和音频信息（控制长度），并用两层 MLP 对齐大模型 hidden layer；冻结编码器，训练 Q-Former 和 MLP：基本都是常规做法。

大模型内部的 LoRA 设计是主要创新点。作者用同一套 LoRA（不是多套 LoRA 分别处理不同任务），但是将 LoRA 的低秩矩阵拆分实现为共用的 A 和多个不同的 B，即共享一部分，独立一部分；同时作者引入 Router R，输出每个 LoRA 头的权重与 LoRA 输出加权融合。

在训练部分，作者仍然是随机抽取不同任务的 batch 进行训练（但是使用 AV-UIE 数据集），loss 也仍然是不同任务的加权和。

DTOS: Dynamic Time Object Sensing with Large Multimodal Model

问题背景

参照视频目标分割（RVOS）即根据文本引导在视频流中检索指定目标并切割的任务，现在主流的做法是视觉-文本多模态大模型。作者在 Abstract 中提出了当前 RVOS 领域的三个问题：

  1. 数值表征不足：RVOS 需要大模型输出时间、位置等信息，而当前的大语言模型虽然擅长聊天，但是对精确数字的处理不好；
  2. 大模型退化：大模型的回复模板固定，造成了模型退化，性能下降（自回归的特性，大模型会认为只要按模板回复就是对的）；
  3. 传统的方法是先采样（随机采样或者均匀采样），然后再把采样帧交给大模型，这种做法的目的是减少大模型输入，但是有可能恰好丢失包含了目标的关键帧。

创新点

为了解决前两个问题，作者从大模型输出模板入手，首先引入 <reg> 标签，用单独的 MLP 处理输出向量得到数值输出，而不是让大模型去查词表；同时给大模型不同的输出模板，丰富输出，避免了大模型盲目自信（认为说这个模板就一定是对的）和重复退化（连续重复几次之后模型开始无限复读）。这部分是相对次要的，重头戏在采样方法上的改进。

为什么要采样？因为让大模型看视频的每一帧是很耗费 token 的，因此我们引入池化、采样、压缩、时空分离编码（先让视觉模型看每一帧，然后把提取出来的特征给大模型看）等方法。

之前的模型采用随机采样或者均匀采样，容易丢失关键帧，锁定到不相关的时间段。现有的 TFS 算法，虽然引入了文本引导采样，但是在实现上是把锁定到的相关的帧抽出来直接拼在一起，丢弃中间的不相关帧，这种做法会导致动作的连续性被破坏，同样不可取。

作者因此设计了 TCS 模块，用大模型先“粗看”一遍视频，输出目标出现的时间段，然后交给 TCD 模块进行精细的目标检测和分割。有人可能会问，我们采样的目的不就是让大模型尽可能少费 token 吗？现在在 TCS 里大模型先看一遍视频，这不是反而看得更多了吗？关键在于实现细节，也就是要“粗看”。

推入 TCS，得到目标可能存在的时间片段。TCD 随后拿到这些片段，先用 Box 在某一帧上框出目标所在的位置，然后生成精细的像素级的 mask，最后用对象跟踪器将一帧的 mask 推送到前后的片段上（这是避免为每一帧重复计算 mask 的常用操作，节约性能）。TCS 和 TCD 都用到了大模型，我们给大模型装不同的 LoRA 即可，如图所示。

上面提到的 <reg> 标签和多个回复模板主要应用于 TCS 的输出上，用 <reg> 标签输出 $[start, end]$ 的时间段数值，以及提供各种不同的回复模板——之前有些模型的输出模板是固定的“目标出现在 1 秒到 3 秒”，这样的模板会误导模型，认为目标只出现了一次，新的模板则诱导模型输出多段时间片段。这样做效果更好，如果目标出现了多次，旧回答模板会遗漏，而新模板不会；如果出现了 1 次，新模板确实会多回答一次目标不存在的时间段，但是这个错误在 TCD 部分可以得到修正：我们更希望不要遗漏。严谨的说，这是把检测的召回率放在 TCS，把准确率交给后续阶段。

在训练上，$L_{TCS}=L_{txt}+L_{reg}$，即文本交叉熵损失和输出时间段的回归损失，$L_{reg}$ 又是 $L_{gIoU}, L_1, L_{label}$ 的加权和。

  • $L_{label}$：判断找到的是不是我们想要的目标，标签是“足球”而找到“篮球”，就会吃到巨大的惩罚；
  • $L_1$：时间点的绝对误差，目标 10 秒出现但模型输出 12 秒，2 秒就是绝对误差；
  • $L_{gIoU}$：广义交并比损失，交并比 IoU 描述两个区间的重叠程度，gIoU 进一步描述了两个区间完全不重叠时的梯度。

TCD 的损失则是 $L_{txt}, L_{box}, L_{sam}$ 的加权和，文本交叉熵损失，框选的损失和切割的损失。
